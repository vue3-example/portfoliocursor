# 커피 주문 앱

## 1. 프로젝트 개요

### 1.1 프로젝트명

커피 주문 앱

### 1.2 프로젝트 목적

사용자가 커피 메뉴를 주문하고, 관리자가 주문을 관리할 수 있는 간단한 풀스택 웹 앱

### 1.3 개발 범위

주문하기 화면(메뉴 선택 및 장바구니 기능)
관리자 화면(재고 관리 및 주문 상태 관리)
데이터를 생성/조회/수정/삭제할 수 있는 기능

## 2. 기술 스택

프런트엔드: HTML, CSS, 리액트, 자바스크립트
백엔드: Node.js, Express
데이터베이스: PostgreSQL

## 3. 기본 사항

프런트엔드와 백엔드를 따로 개발
기본적인 웹 기술만 사용
학습 목적이므로 사용자 인증이나 결제 기능은 제외
메뉴는 커피 메뉴만 있음

## 4. 주문하기 화면 PRD

### 4.1 문서 목적

`주문하기` 화면의 UI 구성, 사용자 플로우, 기능 요구사항, 예외 처리 기준을 정의한다.  
본 문서는 첨부된 와이어프레임을 기준으로 구현 범위를 명확히 하는 데 목적이 있다.

### 4.2 대상 사용자

- 매장에서 커피를 빠르게 주문하려는 일반 사용자
- POS/키오스크 형태의 단순 주문 화면을 사용하는 사용자

### 4.3 화면 목표

- 사용자가 메뉴를 확인하고 옵션을 선택해 장바구니에 담을 수 있어야 한다.
- 장바구니에서 수량/금액을 확인하고 최종 주문을 요청할 수 있어야 한다.
- 화면 구조는 단순하고 직관적이어야 하며, 최소 클릭으로 주문이 가능해야 한다.

### 4.4 화면 구성(와이어프레임 기준)

1. 상단 헤더

- 좌측: 브랜드/앱명(`COZY`)
- 우측: 탭 또는 버튼 2개
  - `주문하기` (현재 활성 상태)
  - `관리자` (관리자 화면 이동)

2. 메뉴 카드 영역(그리드)

- 카드 항목(예시): 아메리카노(ICE), 아메리카노(HOT), 카페라떼
- 카드 구성 요소
  - 메뉴 이미지(없을 경우 플레이스홀더)
  - 메뉴명
  - 기본 가격
  - 짧은 설명
  - 옵션 체크박스
    - 샷 추가 (+500원)
    - 시럽 추가 (+0원)
  - `담기` 버튼

3. 장바구니 영역(하단 패널)

- 장바구니 아이템 목록
  - 주문명(메뉴 + 선택 옵션)
  - 수량
  - 항목별 금액
- 우측 정보
  - 총 금액
  - `주문하기` 버튼

### 4.5 사용자 시나리오

1. 사용자는 주문하기 화면에 진입한다.
2. 메뉴 카드에서 원하는 메뉴를 고른다.
3. 옵션(샷/시럽)을 선택한다.
4. `담기` 버튼을 눌러 장바구니에 추가한다.
5. 같은 메뉴/같은 옵션이면 수량이 증가하고, 옵션이 다르면 별도 항목으로 추가된다.
6. 장바구니에서 총 금액을 확인한 뒤 `주문하기` 버튼을 눌러 주문을 생성한다.

### 4.6 기능 요구사항

#### FR-01. 메뉴 목록 조회 및 노출

- 시스템은 주문 가능한 메뉴 목록을 화면에 카드 형태로 표시해야 한다.
- 각 카드에는 메뉴명, 가격, 설명, 이미지(또는 플레이스홀더)가 포함되어야 한다.

#### FR-02. 옵션 선택

- 사용자는 각 메뉴 카드에서 옵션을 체크박스로 선택할 수 있어야 한다.
- 옵션별 추가 금액이 명확히 표시되어야 한다.

#### FR-03. 장바구니 담기

- 사용자가 `담기` 버튼을 누르면 선택한 메뉴+옵션 조합이 장바구니에 추가되어야 한다.
- 동일 조합이 이미 있으면 해당 항목 수량을 +1 해야 한다.

#### FR-04. 장바구니 표시

- 장바구니에는 항목명, 수량, 항목 금액이 표시되어야 한다.
- 총 금액은 항상 최신 상태로 계산되어야 한다.

#### FR-05. 주문 요청

- 장바구니가 비어있지 않을 때만 `주문하기` 버튼을 활성화한다.
- 사용자가 `주문하기`를 누르면 주문 생성 API를 호출한다.
- 주문 성공 시 장바구니를 비우고 성공 피드백을 제공한다.

#### FR-06. 화면 이동

- `관리자` 버튼을 누르면 관리자 화면으로 이동할 수 있어야 한다.

### 4.7 비즈니스 규칙

- 항목 금액 = (기본 메뉴 가격 + 선택 옵션 추가금 합) x 수량
- 총 금액 = 장바구니 모든 항목 금액 합
- 동일 항목 판별 기준: `menuId + optionSet`
- 옵션 추가금은 서버 응답 값을 기준으로 계산한다(프런트 하드코딩 최소화)

### 4.8 상태 및 예외 처리

- 메뉴 조회 실패: "메뉴를 불러오지 못했습니다. 다시 시도해 주세요."
- 주문 요청 실패: "주문 처리에 실패했습니다. 잠시 후 다시 시도해 주세요."
- 빈 장바구니 주문 시도: 주문 요청 차단 + 안내 메시지 표시
- 네트워크 지연 시: 버튼 로딩 상태 및 중복 클릭 방지

### 4.9 UI/UX 요구사항

- `주문하기` 탭은 활성 상태를 시각적으로 명확히 표시
- 메뉴 카드는 동일한 높이/간격으로 정렬
- 금액은 `#,###원` 형식으로 표시
- 버튼 상태(기본/호버/비활성/로딩)를 구분
- 장바구니 영역은 메뉴 선택 중에도 항상 확인 가능해야 함

### 4.10 접근성 및 품질 기준

- 체크박스/버튼은 키보드 포커스 가능해야 한다.
- 모든 이미지에는 대체 텍스트가 있어야 한다.
- 색상 대비는 텍스트 가독성을 해치지 않아야 한다.
- 주요 상호작용(`담기`, `주문하기`)은 1초 이내 시각적 반응을 제공해야 한다.

### 4.11 수용 기준(AC)

- AC-01: 메뉴 카드가 1개 이상 렌더링된다.
- AC-02: 옵션 선택 후 `담기` 시 장바구니에 정확한 항목명/금액이 추가된다.
- AC-03: 동일 메뉴+옵션을 2회 담으면 수량만 증가한다.
- AC-04: 다른 옵션 조합은 별도 항목으로 분리된다.
- AC-05: 총 금액이 항목 합계와 항상 일치한다.
- AC-06: 장바구니가 비어 있으면 `주문하기` 버튼이 비활성화된다.
- AC-07: 주문 성공 시 성공 메시지 표시 후 장바구니가 초기화된다.
- AC-08: 주문 실패 시 장바구니 데이터는 유지된다.

### 4.12 범위 제외(주문하기 화면 1차)

- 결제(카드/간편결제) 처리
- 로그인/회원 관리
- 쿠폰/포인트/할인 정책
- 복잡한 옵션 조합(예: 사이즈/원두/우유 종류 다중 선택)

## 5. 관리자 화면 PRD

### 5.1 문서 목적

`관리자` 화면의 UI 구성, 운영 플로우, 기능 요구사항, 예외 처리 기준을 정의한다.  
본 문서는 첨부된 관리자 와이어프레임을 기준으로 1차 구현 범위를 명확히 한다.

### 5.2 대상 사용자

- 매장 운영자(점주 또는 직원)
- 주문 처리 및 재고 수량을 확인/조정해야 하는 사용자

### 5.3 화면 목표

- 현재 주문 처리 현황을 한눈에 파악할 수 있어야 한다.
- 메뉴별 재고를 빠르게 증감할 수 있어야 한다.
- 들어온 주문의 상태를 간단히 변경할 수 있어야 한다.

### 5.4 화면 구성(와이어프레임 기준)

1. 상단 헤더

- 좌측: 브랜드/앱명(`COZY`)
- 우측: 탭 또는 버튼 2개
  - `주문하기` (주문 화면 이동)
  - `관리자` (현재 활성 상태)

2. 관리자 대시보드 요약 카드

- 지표 항목
  - 총 주문 수
  - 주문 접수 수
  - 제조 중 수
  - 제조 완료 수

3. 재고 현황 영역

- 메뉴 카드(예시): 아메리카노(ICE), 아메리카노(HOT), 카페라떼
- 카드 구성 요소
  - 메뉴명
  - 현재 재고 수량(예: 10개)
  - `+` 버튼(재고 증가)
  - `-` 버튼(재고 감소)

4. 주문 현황 영역

- 주문 리스트(행 단위)
  - 주문 시각
  - 주문 항목 요약(메뉴/수량)
  - 주문 금액
  - 상태 변경 버튼(예: `주문 접수`)

### 5.5 사용자 시나리오

1. 관리자는 관리자 화면에 진입한다.
2. 대시보드에서 현재 주문 상태별 건수를 확인한다.
3. 재고 현황에서 특정 메뉴 재고를 `+/-` 버튼으로 조정한다.
4. 주문 현황에서 주문 상태 버튼을 눌러 다음 단계로 변경한다(예: 주문 접수 -> 제조 중 -> 제조 완료).
5. 상태가 변경되면 주문 목록과 대시보드 지표가 즉시 갱신된다.

### 5.6 기능 요구사항

#### FR-101. 관리자 대시보드 조회

- 시스템은 주문 상태별 집계값을 조회하여 화면에 표시해야 한다.
- 집계값은 주문 상태 변경 후 최신 값으로 갱신되어야 한다.

#### FR-102. 재고 목록 조회 및 표시

- 시스템은 메뉴별 현재 재고 수량을 카드 형태로 표시해야 한다.
- 재고 값은 0 이상의 정수로 표시되어야 한다.

#### FR-103. 재고 증감 처리

- 사용자는 `+` 버튼으로 재고를 1 증가시킬 수 있어야 한다.
- 사용자는 `-` 버튼으로 재고를 1 감소시킬 수 있어야 한다.
- 재고가 0일 때 `-` 동작은 차단되어야 한다.

#### FR-104. 주문 목록 조회

- 시스템은 최신 주문 목록을 시간순으로 표시해야 한다.
- 각 주문 행에는 주문 시각, 주문 요약, 금액, 현재 상태가 표시되어야 한다.

#### FR-105. 주문 상태 변경

- 관리자는 주문 상태를 단계적으로 변경할 수 있어야 한다.
- 기본 상태 흐름은 `주문 접수 -> 제조 중 -> 제조 완료`를 따른다.
- 상태 변경 성공 시 목록과 대시보드 지표를 갱신해야 한다.

#### FR-106. 화면 이동

- `주문하기` 버튼을 누르면 주문 화면으로 이동할 수 있어야 한다.

### 5.7 비즈니스 규칙

- 주문 상태는 정의된 단계 순서대로만 변경 가능하다.
- `제조 완료` 상태 주문은 추가 상태 변경이 불가능하다.
- 재고 수량의 최소값은 0이며 음수 저장은 허용하지 않는다.
- 재고 변경/주문 상태 변경은 서버 저장 성공 이후 화면에 확정 반영한다.

### 5.8 상태 및 예외 처리

- 대시보드/목록 조회 실패: "관리자 데이터를 불러오지 못했습니다. 다시 시도해 주세요."
- 재고 변경 실패: "재고 변경에 실패했습니다. 잠시 후 다시 시도해 주세요."
- 상태 변경 실패: "주문 상태 변경에 실패했습니다. 다시 시도해 주세요."
- 동시 수정 충돌(이미 다른 사용자가 변경): 최신 데이터 재조회 후 안내 메시지 표시
- 요청 처리 중 동일 버튼 연속 클릭 방지(로딩/비활성 상태)

### 5.9 UI/UX 요구사항

- `관리자` 탭은 활성 상태를 시각적으로 명확히 표시
- 대시보드, 재고, 주문 현황 섹션은 구분된 카드/패널 형태로 배치
- 재고 증감 버튼은 충분한 클릭 영역을 제공
- 주문 상태 버튼은 현재 상태를 명확히 인지할 수 있어야 함
- 금액/수량 표시는 일관된 형식(`#,###원`, `N개`)을 사용

### 5.10 접근성 및 품질 기준

- 모든 버튼(`+`, `-`, 상태 변경)은 키보드 포커스 및 Enter/Space 조작이 가능해야 한다.
- 상태 텍스트는 색상 외에도 텍스트 자체로 구분 가능해야 한다.
- 화면 주요 작업(재고 변경, 상태 변경)은 사용자 액션 후 1초 이내 피드백을 제공해야 한다.
- 로딩/에러 메시지는 사용자가 인지 가능한 위치에 표시되어야 한다.

### 5.11 수용 기준(AC)

- AC-101: 관리자 진입 시 대시보드 지표 4개가 표시된다.
- AC-102: 메뉴별 재고 카드가 1개 이상 렌더링된다.
- AC-103: `+` 클릭 시 해당 메뉴 재고가 1 증가한다.
- AC-104: 재고가 0일 때 `-` 클릭은 반영되지 않고 안내 또는 비활성 상태를 유지한다.
- AC-105: 주문 목록에 주문 시각/요약/금액/상태가 표시된다.
- AC-106: 주문 상태 버튼 클릭 시 다음 단계 상태로 변경된다.
- AC-107: `제조 완료` 주문은 상태 변경 버튼이 비활성 또는 숨김 처리된다.
- AC-108: 상태 변경 후 대시보드 집계값이 즉시 일관되게 갱신된다.

### 5.12 범위 제외(관리자 화면 1차)

- 권한별 접근 제어(매니저/직원 역할 분리)
- 주문 취소/환불 처리
- 재고 이력(감사 로그) 상세 조회
- 다중 매장 관리, 알림/푸시, 통계 차트 고도화

## 6. 백엔드 PRD (데이터 모델 및 API)

### 6.1 데이터 모델 개요

- **Menus**: 커피 메뉴 정보 및 재고
  - `id` (PK, UUID 또는 bigserial): 메뉴 식별자
  - `name` (varchar, not null): 커피 이름
  - `description` (text, nullable): 설명
  - `price` (integer, not null): 기본 가격(원 단위)
  - `image_url` (text, nullable): 이미지 URL
  - `stock_quantity` (integer, not null, default 0): 현재 재고 개수
  - `created_at` (timestamptz, not null)
  - `updated_at` (timestamptz, not null)
- **Options**: 메뉴별 옵션 정보
  - `id` (PK)
  - `menu_id` (FK → Menus.id, not null): 연결할 메뉴
  - `name` (varchar, not null): 옵션 이름 (예: 샷 추가)
  - `extra_price` (integer, not null, default 0): 옵션 추가 금액
  - `created_at` (timestamptz, not null)
  - `updated_at` (timestamptz, not null)
- **Orders**: 주문 헤더
  - `id` (PK)
  - `ordered_at` (timestamptz, not null, default now()): 주문 일시
  - `status` (varchar, not null): 주문 상태
    - 값: `RECEIVED`(주문 접수), `IN_PROGRESS`(제조 중), `COMPLETED`(완료)
  - `total_price` (integer, not null): 주문 총 금액
  - `created_at` (timestamptz, not null)
  - `updated_at` (timestamptz, not null)
- **OrderItems**: 주문 상세(메뉴/옵션/수량)
  - `id` (PK)
  - `order_id` (FK → Orders.id, not null)
  - `menu_id` (FK → Menus.id, not null)
  - `menu_name_snapshot` (varchar, not null): 주문 시점 메뉴 이름
  - `unit_price` (integer, not null): 1잔 가격(메뉴 + 옵션)
  - `quantity` (integer, not null)
  - `line_price` (integer, not null): `unit_price * quantity`
  - `option_names_snapshot` (text, nullable): 선택 옵션 이름 목록 문자열
  - `created_at` (timestamptz, not null)

### 6.2 데이터 흐름 (사용자 플로우 기준)

1. **메뉴 조회**
   - 서버는 Menus(+Options)를 읽어 주문 화면/관리자 화면에 전달한다.
   - 주문 화면: 이름, 설명, 가격, 이미지, 옵션 정보만 사용.
   - 관리자 화면: Menus의 `stock_quantity`로 재고 상태를 표시한다.
2. **장바구니 구성**
   - 사용자가 앱 화면에서 커피 메뉴와 옵션을 선택해 장바구니에 담는다.
   - 이 단계에서는 서버에 저장하지 않고, 프런트 상태로만 유지해도 된다.
3. **주문 생성**
   - 사용자가 `주문하기` 버튼을 누르면, 프런트는 장바구니(메뉴, 수량, 옵션, 예상 금액)를 주문 생성 API로 보낸다.
   - 서버는 Menus/Options를 기준으로 **실제 금액을 재계산**하고,
     - 메뉴별 주문 수량 합계가 Menus.stock_quantity보다 크면 **주문 실패**로 응답한다.
     - 아니면 Orders + OrderItems에 저장하고 Menus.stock_quantity를 수량만큼 차감한다.
   - Orders의 초기 상태는 항상 `RECEIVED`이다.
4. **주문 현황 표시**
   - 관리자 화면 진입 시 서버에서 Orders(+OrderItems 요약)를 조회해 `주문 현황`에 보여 준다.
   - 표시 내용: 주문 일시, 주문 내용 요약(메뉴/옵션/수량), 금액, 상태.
5. **주문 상태 변경**
   - 관리자가 주문 상태 변경 버튼을 누르면 상태 변경 API를 호출한다.
   - 상태 흐름: `RECEIVED → IN_PROGRESS → COMPLETED` 순서만 허용한다.
   - 상태 변경 후 Orders 상태가 갱신되고, 관리자 화면의 주문 현황과 대시보드 집계가 갱신된다.

### 6.3 API 설계 개요

1. **메뉴 목록 조회 API**
   - 메서드/URL: `GET /api/menus`
   - 역할: Menus + Options를 조회하여
     - 주문 화면: 이름/설명/가격/이미지/옵션
     - 관리자 화면: 재고 수량까지 포함해 내려준다.
2. **주문 생성 API**
   - 메서드/URL: `POST /api/orders`
   - 요청 바디(예시 개념):
     - `items[]`: `{ menuId, quantity, optionIds[] }` 배열
   - 처리:
     - Menus/Options로부터 실제 가격 계산
     - 재고 부족 시 `400 Bad Request` + 부족 메뉴 정보 반환
     - 성공 시 Orders + OrderItems 생성, Menus.stock_quantity 차감
   - 응답:
     - 생성된 주문 ID, 주문 시간, 총 금액, 상태(`RECEIVED`)
3. **주문 목록 조회 API**
   - 메서드/URL: `GET /api/orders`
   - 옵션: `status`(필터), `limit/offset`(선택)
   - 역할: 관리자 화면 `주문 현황`에 필요한 주문 리스트(주문 일시, 요약 문자열, 금액, 상태)를 제공한다.
4. **단일 주문 상세 조회 API**
   - 메서드/URL: `GET /api/orders/:orderId`
   - 역할: 특정 주문의 헤더 + 상세(메뉴/수량/옵션/금액)를 반환해 상세 팝업/디버깅 등에 사용한다.
5. **주문 상태 변경 API**
   - 메서드/URL: `PATCH /api/orders/:orderId/status`
   - 요청 바디: `nextStatus` (`IN_PROGRESS` 또는 `COMPLETED`)
   - 규칙:
     - `RECEIVED → IN_PROGRESS`, `IN_PROGRESS → COMPLETED` 만 허용.
     - 그 외 전이는 400 에러로 거부.
6. **재고 수정 API (관리자)**
   - 메서드/URL: `PATCH /api/menus/:menuId/stock`
   - 요청 바디: `delta` (정수, 예: `+1` / `-1`)
   - 역할: 관리자 화면 재고 카드의 `+ / -` 버튼과 연결해 Menus.stock_quantity를 증감시키며, 0 미만은 허용하지 않는다.
